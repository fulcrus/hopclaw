# Runbook: 灾备与故障接管

## 1. 当前官方支持的灾备语义

HopClaw 当前推荐的灾备模型是：

- 单活实例
- 备份恢复
- 手工或平台编排切换

不是：

- 共享状态多活
- 自动主从选主
- 内建跨实例共识

## 2. 建议目标

对大多数团队，先给出可落地目标：

- RPO: 24h 内
- RTO: 30min 到 2h

如果你们已经有更高要求：

- 用更密集备份
- 用平台级卷快照
- 用外围业务入口控制切流

## 3. 典型故障场景

### 节点损坏

- 原机不可用
- 状态目录仍可从备份恢复

### 磁盘损坏或误删

- 直接从最近备份恢复
- 检查 `.bak` 文件辅助排查

### 企业外部依赖异常

- 模型平台故障
- 审批系统故障
- 审计接收端故障

这类场景不一定需要切换 HopClaw 节点，优先做外部依赖绕行或 redrive。

## 4. 冷恢复流程

1. 在新节点准备同版本 HopClaw
2. 挂载新的空状态目录
3. 执行 `hopclaw backup restore <archive>`
4. 启动 HopClaw
5. 检查 `/healthz`
6. 检查控制面状态和 audit delivery stats
7. 放开流量

## 5. 热备边界

如果企业平台自己提供：

- 块存储快照
- VM / 容器编排恢复
- DNS / LB 切换

那么可以把 HopClaw 放进你们的统一灾备流程里。

但官方仍建议：

- 一个状态目录只让一个活跃 HopClaw 写入

## 6. 切换后验证

至少验证：

- 健康检查正常
- 最近 session 可读
- 新 run 可以创建
- 审计 sink 可以投递
- dead-letter 没有继续堆积

## 7. 建议演练频率

- 季度一次完整灾备演练
- 月度一次恢复演练

没有演练，就不要把“灾备”写成既成事实。
